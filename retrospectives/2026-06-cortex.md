# Retrospective: Cortex

**Dates**: 2026-06-13 → 2026-06-19
**Skills package version used**: v0.2.0 (pm-checklist was added mid-project)
**Outcome**: shipped (CLI tool `lt`, 52 tests, all 4 spec phases complete, in daily use)
**Stack**: Python 3.11+, OpenAI (text-embedding-3-small + gpt-4o-mini), sqlite-vec, Click, pytest+respx

---

## TL;DR

Built a personal RAG CLI that ingests a notes folder and answers questions with cited sources. The multi-agent pipeline (Opus plan → DeepSeek V4 Pro implement → Opus review → fix) worked end-to-end across 4 phases for ~$0.05 of implementer cost. Biggest lesson: **the implementer hallucinated function names from neighboring modules in every single phase** — the planner prompts described intent but not the actual call signatures in scope. The fix is structural, not "be more careful."

---

## What worked

- **`spec-first`** — the spec carried 4 phases without contradicting itself. When Phase 3 review found the squared-L2 similarity bug, the fix was a spec lookup ("§7 says cosine similarity for OpenAI unit vectors"), not a re-argument.
- **`pm-checklist`** (added v0.2.0 mid-project) — Section 5 (scope) and Section 9 (non-negotiables) took the user 5 minutes each and prevented multiple rounds of "wait, should this be a feature?" later. User said unprompted post-Phase 1: *"I really like the pm-checklist, it makes decisions more clear and less convoluted."* Held up through Phases 2–4.
- **`adversarial-review` (Opus)** — caught two real ship-blockers:
  - Phase 3: `similarity = 1.0 - distance` (wrong — sqlite-vec returns *squared* L2 by default; correct formula is `1 - distance/2` for unit vectors)
  - Phase 3: source-attribution invariant enforced with `assert`, strippable under `python -O`; promoted to `RuntimeError`
- **`non-negotiable-paths`** — the spec's source-attribution rule ("every `ok` answer must cite ≥1 source") got encoded as a post-condition that raises on violation. Three weeks from now I won't remember the rule, but the code will refuse to ship without it.
- **Multi-model split** — Opus's blind spots (verbose plans) and DeepSeek V4 Pro's blind spots (existing-name hallucination, silent truncation) genuinely don't overlap. Same model planning and reviewing would have missed the squared-L2 bug because the planner wrote the wrong formula.
- **`ds-send` direct curl wrapper** replaced a heavier Python-venv wrapper from the previous project. 130 lines of bash+jq, no dep on llm-gateway. The user's pushback on the orchestrator wrapper ("can you do it without the wrapper?") was correct — the wrapper was solving a problem that didn't exist in this context.

## What failed

- **Implementer hallucinated function names across all 3 implementation phases.** Phase 2: called `init_db(conn)` when `init_db(path)` is the actual signature. Phase 3: same `init_db` confusion in `retrieve()`. Phase 4: imported `chunk_file` and `walk` — both nonexistent (actual names: `chunk_text`, `walk_sources`). Root cause: the planner described intent ("call the chunker on each file") but not the in-scope function signatures. DeepSeek V4 Pro plausibly guessed names that fit the description.
- **DeepSeek V4 Pro silently truncated at `max-tokens 16384`** for multi-file output in Phase 2. Thinking mode burns ~24K tokens before any user-visible content, so the budget was already gone before code started. Bumped to 65536 and it succeeded. No warning, no error — output just stopped. Same failure mode as llm-gateway Phase 6.
- **respx `assert_all_called=True` default** caused a Phase 3 test to fail spuriously because the test deliberately expected one mocked route to be uncalled. Needed `respx.mock(assert_all_called=False)`. Hit again in Phase 4. Both times the implementer didn't know the default.
- **Phase 4 test bugs (5 in one phase)**: `ingest_directory(db_path=...)` with no such kwarg; `route.calls == 0` instead of `len(route.calls) == 0`; missing `import pytest` in test_config.py; absolute-vs-relative path assertion too strict; `KeyError` double-quoting in error message. All caused by implementer pattern-matching on superficially similar test idioms without checking the actual API.
- **`lt ask` quoting papercut** shipped with v1.0.0. Click's default `@click.argument("query")` treats `lt ask what did I write` as 5 args instead of one question. User hit this on their second real query. Fixed post-ship today (v1.0.1 candidate): `nargs=-1` + `" ".join()`.
- **Live API key pasted into the conversation transcript** during shell setup. Operational hygiene failure — the key is now in the conversation log AND `~/.bashrc`. Not a skill failure per se, but the env-var pattern needs a more obvious "load from file, don't paste" path for first-time setup.

## Friction log review

- **[missing-skill]** Implementer doesn't get a "names in scope" list. Three phases of hallucinated function names is signal, not noise. The planner prompt should enumerate every importable name the implementer needs to use, with signatures. This is exactly the kind of thing that's cheap at plan-time and expensive at review-time.
- **[missing-skill]** No "embedding-store gotchas" checklist. sqlite-vec returns *squared* L2 by default — not documented prominently, not in any prompt, caught only by adversarial review. Different vector stores (Pinecone, Qdrant, pgvector, sqlite-vec) all have subtly different distance/similarity conventions. A pre-implementation checklist would have flagged it.
- **[vague-skill]** `adversarial-review` doesn't currently enumerate "any numerical formula derived from a third-party library's distance metric" as a high-priority review target. Adding it would have moved the squared-L2 catch from "lucky" to "structural."
- **[missing-skill]** No "library-default-values" awareness for test code. respx defaults to `assert_all_called=True`. pytest fixtures default to function scope. These trip up implementers who pattern-match on test shapes from training data without checking the version's behavior.
- **[missing-skill]** No "non-negotiable enforcement uses RuntimeError, not assert" rule. The codebase's source-attribution invariant was originally written as `assert`, which is silently stripped under `python -O`. Caught by Opus review. The fix is a one-liner skill.

## Skills changes proposed

### New skills

- **`engineering/implementer-handoff`** — Before invoking the implementer model, the plan/handoff prompt must include:
  - **Names in scope** — every importable function/class/method the implementer needs, with current signatures. *Source: 3 phases of hallucinated function names in Cortex (`init_db(conn)` × 2, `chunk_file`, `walk`).*
  - **Library gotchas** — relevant defaults and conventions for libraries being used (sqlite-vec returns squared L2 not cosine; respx defaults to `assert_all_called=True`; pytest fixtures default to function scope). Grows organically as new gotchas are discovered. *Source: squared-L2 similarity bug + 2 respx test bugs.*
  - **Output budget** — `max-tokens` floor of 65536 for thinking-mode models (DeepSeek V4 Pro, o-series). Thinking burns ~24K tokens silently before content. *Source: Phase 2 silent truncation + llm-gateway Phase 6 truncation (2-incident pattern).*

  Single trigger (about to call implementer), single action (extend handoff prompt to cover these bases), single file. Future incidents become new bullets, not new files.

### Modify existing

- **`workflow/pm-checklist`** — Validated through Phases 2–4. Two refinements: (1) Phase-N checklists should be visibly shorter than Phase-1 (less to decide later in a project); (2) include a "carry-forward" section explicitly listing decisions from prior phases that still apply, to prevent the implementer from re-litigating them.
- **`quality/non-negotiable-paths`** — Add: *When encoding a non-negotiable invariant in code, use `raise RuntimeError(...)`, not `assert`. Asserts are stripped under `python -O`, silently removing the invariant in production builds.* Source: Phase 3 adversarial review found the source-attribution invariant written as `assert` and promoted it. One-paragraph addition to an existing skill, not a new file.
- **`quality/adversarial-review`** — Add explicit review patterns: any numerical formula derived from a third-party library's metric (distance, similarity, p-value, etc.); any `assert` in non-test code; any positional Click argument that might receive multi-word user input.

### Delete

- None this project. The skills v0.2.0 set all earned their keep.

An earlier draft of this section proposed four new skills. Reviewed and consolidated to one. ## Insights worth keeping (beyond skill changes)

These aren't skills — they're durable observations about the tooling, worth re-reading before the next RAG project.

1. **`text-embedding-3-small` runs cool.** A chunk that perfectly answers a question often scores 0.3–0.5, not 0.7+. Questions and answers have different "shape" in embedding space (asymmetric retrieval). Calibrate `min_similarity` empirically against your own corpus, not by published rubrics. Defaults should probably be 0.2–0.25 for personal note corpora.

2. **Citations matter more than scores.** The non-negotiable that every answer cites ≥1 source is the user-trust mechanism. Scores are debugging signal, not quality gate. Surfacing scores too prominently anchors user expectations on a noisy number.

3. **Per-file transaction atomicity for ingest** turned out to matter — one corrupt file shouldn't roll back the whole batch. Idempotency via `content_hash` makes re-running cheap. Both decisions were in the spec from Day 1 and never needed to be revisited.

4. **The user's "without a wrapper?" instinct was right.** When evaluating whether to reuse infrastructure from a prior project, default to "simplest thing that calls the API directly" before reaching for the previous abstraction. ds-send (130 lines bash) replaced lg-send (a Python wrapper around the llm-gateway CLI) with zero loss in capability for this use case.

5. **DeepSeek V4 Pro implementer cost was ~$0.05 across 3 phases.** That's the price ceiling on "implementer correctness improvements" — if a skill costs more agent-time than that to apply, the math doesn't work for solo projects.

6. **Skill granularity — watch the over-split trap.** When extracting skills from a retrospective, the temptation is to make each incident into its own skill. Test: do they share a trigger AND an action? If yes, they're one skill with multiple bullets, not multiple skills. Three of the four originally-proposed v0.3.0 skills failed this test — same trigger ("about to call implementer"), same action ("extend handoff prompt"). Over-splitting looks more thorough but fragments the underlying lesson and creates discovery problems (which file fires when? answer: all of them, every time — so it's really one file).

- Will `engineering/implementer-handoff` actually fix the name-hallucination problem, or will the implementer still grab the wrong name from training data even when shown the right one in the "names in scope" block? Worth designing the prompt format adversarially — does the implementer obey when contradicted, or does it confabulate?
- Does the consolidated handoff skill scale, or does it accumulate bullets faster than it accumulates utility? Re-evaluate at the next end-of-project retro: is the skill still one cohesive read, or has it become a list nobody reads?
- The squared-L2 bug suggests that any time the implementer writes a *formula* involving a library return value, the review should re-derive the formula from first principles. Worth formalizing as a sub-pattern under `adversarial-review`.
